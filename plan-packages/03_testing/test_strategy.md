## 1. TDD 原则

核心解析能力必须先写测试再实现：

- 日志行解析。
- 时间窗口过滤。
- PID 生命周期追踪。
- 多包名合并。
- 动态 Tag 统计。
- 系统关键事件抽取。
- AI 报告输入结构生成。
- AI Provider factory、mock provider 和 DeepSeek provider。
- Web 本地后端健康检查。

每个行为至少经历：

```text
RED：写失败测试
GREEN：最小实现通过测试
REFACTOR：整理结构并保持测试通过
```

## 2. 样例测试场景

### T1 单包名单 PID

输入一个包名和一个窗口，日志中只有一个 PID。

预期：

- 识别唯一 PID。
- 裁剪目标 PID 日志。
- 输出动态 Tag 统计。

### T2 单包名多 PID 重启

日志中包含启动、死亡、重启。

预期：

- 输出多个生命周期片段。
- 不混淆旧 PID 和新 PID。

### T3 多包名跨进程

输入两个包名，两个进程在同一窗口内都有异常日志。

预期：

- 分别输出 PID 生命周期。
- 合并跨进程时间线。

### T4 系统关键事件

目标 PID 没有 FATAL，但系统日志存在 ANR 或 timeout。

预期：

- 系统事件纳入时间线。
- 报告标明事件来源。

### T5 异常输入

缺少包名、时间格式错误、找不到 PID。

预期：

- 输出明确错误。
- 不生成伪分诊结论。

### T6 AI Provider 配置

输入：

- `AI_PROVIDER=mock`。
- 无 API key。

预期：

- Demo 可生成稳定 mock 报告。
- 不发起真实外部请求。

输入：

- `AI_PROVIDER=deepseek`。
- 缺少 `AI_API_KEY`。

预期：

- 返回明确配置错误。
- 不静默 fallback 到真实模型。

## 3. AI 报告验收

AI 报告必须：

- 引用 evidence id 或日志行号。
- 区分事实和推测。
- 给出可执行排查路径。
- 对信息不足给出补充信息清单。
- 生成可复制到 Jira 的摘要。

## 4. Verification Before Completion

任何实现完成声明前必须提供新鲜验证证据：

- 测试命令。
- 通过/失败数量。
- 失败时的实际错误。
- 未能运行验证的原因。

禁止使用“应该可以”“看起来没问题”作为完成依据。

## 5. Demo 验收

Demo 必须展示：

- 输入：日志、时间窗口、多个包名。
- 中间结果：PID 生命周期、动态 Tag、关键事件。
- 输出：AI 分诊报告。
- 闭环：HM 自动上传日志、自动建 Jira、AI 回填报告。
- AI：mock provider 可跑通；DeepSeek provider 缺 key 时错误可读；前端不包含 API key。
- 人工验证：每个 Job 的验证条目均可由非 Web 专业开发者按步骤执行。
