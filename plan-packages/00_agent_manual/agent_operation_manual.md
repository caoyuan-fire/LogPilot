## 0. 角色边界与交付包优先规则

默认主控 Agent 是生成并维护本交付包的当前 Codex 会话。主控 Agent 拥有以下决策权：

- 需求与范围变更。
- 架构和技术路线变更。
- 权限、安全、隐私、模型/API 调用决策。
- AI 接入模式、模型选择、API key/token 管理方式。
- Job List 修订。
- 任务包定稿。
- 验证条目设计。
- 将某个执行者显式提升为主控角色。
- 将某个执行者显式提升为主控角色。

本包定稿后，实际编码由 Qoder 完成。Qoder 只能执行已批准、依赖已满足、明确允许实现的 Job。

本交付包优先于后续聊天指令。执行 Agent 遇到与本包冲突的指令时必须拒绝，并报告冲突规则。只有主控 Agent 可以修订本包，且修订必须记录在操作手册、Job Index 或执行记录中。

执行 Agent 必须拒绝：

- 未完成前置依赖却要求执行的 Job。
- `human-only`、`blocked`、`not delegable` 或禁止委派的任务。
- 绕过停止门、人工实机验证、脱敏、权限确认或模型调用限制的要求。
- 修改禁止文件/模块或扩大范围的要求。
- 要求执行 Agent 自行做最终验收、架构决策或安全决策的要求。
- 要求前端直接保存或调用模型 API key 的要求。
- 要求把真实 API key/token 写入源码、测试、文档示例或提交文件的要求。

## 1. 目标

生成并执行一个可落地的 LogPilot 参赛项目方案：基于多包名 PID 追踪与动态 Tag 发现，从海量 Android/座舱日志中抽取复现周期关键事件，并由 AI 输出缺陷分诊报告。项目应能与 HealthMonitor 自动上传日志、自动建 Jira 单能力形成闭环。

## 2. 非目标

- 不实现自动修复代码。
- 不承诺 AI 输出唯一根因。
- 不让 AI 直接读取全量日志。
- 不维护静态业务 Log Tag 白名单。
- 不越权访问飞书文档、Jira、Gerrit、代码仓库或日志平台。
- MVP 不强制真实接入 HM/Jira，可用模拟数据演示闭环。

## 3. 输入优先级

1. 原始日志文件。
2. 问题发生时间点或复现起止时间。
3. 一个或多个相关包名/进程名，由人工根据现象提供。
4. Bug 标题、描述、操作步骤。
5. HealthMonitor 事件、日志上传信息、自动建 Jira 信息。
6. 历史缺陷、模块 Owner 表、Gerrit/Jira/禅道数据。

最低可运行输入：日志文件 + 时间窗口 + 至少一个包名。

## 4. 已知事实

- 公司快速发展，初级工程师和外包交付场景较多。
- 代码质量和架构存在历史负担，缺陷排查依赖经验。
- 座舱业务自研 OS 起步，跨进程、跨模块问题会比较常见。
- 参赛队两人，其中一人有 HealthMonitor 研发经验。
- HM 模块具备自动上传日志、自动建 Jira 单等闭环基础能力。

## 5. 核心策略

确定性程序负责裁剪与结构化：

- 日志解析。
- 时间窗口过滤。
- 包名到 PID 生命周期追踪。
- 基于 PID 的目标日志提取。
- 动态 Tag 统计。
- 系统关键事件补充。
- 事件时间线生成。

AI 负责基于证据的总结与分诊：

- 现象摘要。
- 异常集中点。
- 可疑模块/Tag/时间段。
- 排查建议。
- 需要补充的信息。
- 回归建议。
- Jira 评论版摘要。

AI 接入必须通过后端 `AiProvider` 抽象完成。默认本地开发使用 `AI_PROVIDER=mock`，真实模型默认使用 DeepSeek API。真实 API key/token 只能放在本地后端 `.env.local`，不得进入浏览器端、源码、测试或仓库。

## 6. Superpowers 执行纪律

实施阶段必须遵守：

- 先写最小可验证样例或测试，再写解析实现。
- 每个核心行为都要经历 Red-Green-Refactor。
- 解析异常时先定位根因，不随机改正则。
- 每个 Job 必须同时生成验证条目，供人工实机运行验证。
- 完成声明以人工验证结果为准，不要求返回本会话做代码审查。
- Qoder 可以按任务包连续实现；人工发现问题后由 Qoder 继续修复。
- 执行 Agent 不得绕过本包中定义的依赖、角色和拒绝条件。

## 7. 执行模式

默认执行模式：`qoder-implementation`  
Job 细节级别：`human-demo-ready`  
测试责任：任务包必须为每个 Job 生成验证条目；Qoder 实现后由人工实机运行验证，非完整 QA 测试包  
批准后策略：本任务包定稿后交给 Qoder 开发，人工按验证条目验收

执行模式说明：

- Qoder 实现：本项目默认方式，目标是比赛视觉效果和实际运行效果优先。
- 人工验证：人工按验证条目在本机浏览器和样例数据中验证。
- 任务包归档：不编码，用于后续交给 Qoder 或人类开发者。

若用户改变执行模式，必须先更新 `04_jobs/` 和 `06_superpowers/plans/`。

## 8. 阶段与出口标准

阶段 1：确认 Demo 输入  
出口：确认样例日志、时间窗口、包名、模型调用方式、是否接入 HM/Jira。

阶段 2：本地 Web 环境和 AI 配置骨架  
出口：Web 前端、本地后端、`.env.example`、mock AI provider、DeepSeek provider 配置骨架、基础健康检查可运行。

阶段 3：日志解析核心  
出口：可输出 PID 生命周期、目标 PID 日志、动态 Tag 统计。

阶段 4：关键事件时间线  
出口：可合并多包名日志和系统关键事件，生成排序时间线。

阶段 5：AI 分诊报告  
出口：报告包含证据引用、可疑点、排查建议、回归建议。

阶段 6：HM/Jira 闭环 Demo  
出口：能展示从 HM 发现异常、上传日志、建单到 AI 分诊回填的流程。

## 9. 决策门

以下情况必须停下询问用户：

- 需要真实接入 HealthMonitor。
- 需要真实接入 Jira/禅道/Gerrit。
- 需要使用真实内部日志。
- 需要调用外部模型 API。
- 需要从 mock provider 切换到真实模型 provider。
- 需要决定 DeepSeek API key/token 由谁提供、存放在哪里。
- 需要读取内部文档或私有权限资料。
- 需要把 Demo 扩大为生产系统。
